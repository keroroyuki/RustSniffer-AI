# 产品需求文档 (PRD)：Rust 智能抓包与流量分析工具 (代号：RustSniffer-AI)

## 1. 文档概述
### 1.1 背景与目标
传统的抓包工具（如 tcpdump、Wireshark）存在学习门槛高、缺乏语义理解、难以应对加密流量等痛点。本项目旨在利用 Rust 开发一款**高性能、支持深度包检测（DPI）、并原生集成 AI 大模型能力**的下一代网络流量分析原型工具。
### 1.2 目标受众
*   网络工程师 / 运维人员（需要快速排查网络故障）。
*   安全研究员 / 渗透测试人员（需要分析恶意流量）。
*   对网络底层和 AI 结合感兴趣的开发者。

---

## 2. 系统架构设计
系统采用 **“高性能采集 -> 异步处理 -> 智能分析”** 的解耦架构，充分利用 Rust 的并发优势。

1.  **采集层 (Capture Layer)**：负责底层网卡抓包，要求极致性能，零拷贝。
2.  **处理层 (Processing Layer)**：协议栈解析、TCP 流重组、DPI 精确分类。
3.  **缓冲与存储层 (Buffer & Storage)**：通过无锁队列缓冲，异步写入元数据库和原始包文件。
4.  **AI 分析层 (AI & LLM Layer)**：消费元数据，进行特征提取、自然语言交互、异常检测。
5.  **交互层 (Interaction Layer)**：提供 CLI（命令行）和 TUI（终端用户界面）。

---

## 3. 核心功能需求 (Functional Requirements)

### 3.1 数据采集与基础过滤
*   **网卡选择**：自动枚举系统可用网卡，支持指定网卡抓包。
*   **BPF 过滤**：支持标准的 Berkeley Packet Filter (BPF) 语法进行底层过滤（如 `port 80`）。
*   **抓包模式**：支持混杂模式（Promiscuous mode）和常规模式。

### 3.2 协议解析与精确分类 (DPI)
*   **基础解析**：解析 Ethernet, IPv4/IPv6, TCP/UDP, ICMP 等基础协议。
*   **流重组**：支持 TCP 会话跟踪与流重组（处理乱序、重传）。
*   **深度包检测 (DPI)**：
    *   不仅基于端口，需基于 Payload 特征识别应用层协议（HTTP, DNS, TLS, SSH, 甚至特定的 P2P/游戏协议）。
    *   提取 TLS 握手的 JA3/JA3S 指纹。
*   **元数据提取**：提取五元组、包长、时间戳、方向、应用层关键字段（如 HTTP Host, DNS Query）。

### 3.3 数据存储与检索
*   **元数据存储**：将解析后的结构化数据（非 raw bytes）存入本地数据库，支持快速聚合查询。
*   **原始包存储**：将原始 Pcap 数据按时间或大小切片追加写入本地文件（PCAPNG 格式），支持通过元数据索引回溯原始包。

### 3.4 AI 智能分析 (核心亮点)
*   **自然语言转 BPF/SQL (Text-to-Query)**：
    *   用户输入自然语言（如：“*找出过去 5 分钟内，访问次数最多的前 10 个外部 IP*”）。
    *   AI 将其转化为底层的 BPF 过滤表达式或数据库查询 SQL，并自动执行返回结果。
*   **流量语义总结**：
    *   对特定会话或时间段的流量元数据进行摘要（如：“*该 IP 主要在进行 DNS 查询和 HTTPS 访问，未发现明显的横向移动特征*”）。
*   **异常行为检测 (AI 辅助)**：
    *   基于流的统计特征（包长序列、时间间隔），利用轻量级 ML 模型或 LLM 识别异常（如 DDoS、端口扫描、C2 心跳）。
*   **安全事件解释**：
    *   当检测到可疑流量（如 SQL 注入特征）时，AI 自动解释攻击原理并给出修复建议。

## 4. 技术选型 (Rust 生态)

这是本项目的核心，充分利用 Rust  crates 生态：

| 模块 | 技术/Crate 选型 | 说明 |
| :--- | :--- | :--- |
| **底层抓包** | `pcap` (libpcap 绑定) / `libbpf-rs` (eBPF) | 原型阶段使用 `pcap` 保证跨平台和开发速度；进阶追求极致性能可引入 eBPF/XDP。 |
| **协议解析** | `etherparse` / `pnet` | `etherparse` 是纯 Rust 实现的高性能网络包解析库，无外部 C 依赖。 |
| **DPI 引擎** | `ndpi` (通过 `bindgen` FFI 绑定) | nDPI 是最强的开源 DPI 库。通过 Rust FFI 调用其 C 代码，外层封装安全的 Rust API。 |
| **异步与并发** | `tokio` |  Rust 异步生态事实标准，用于处理高并发的网络 I/O 和 AI 请求。 |
| **内存缓冲** | `crossbeam` / `tokio::sync::mpsc` | 用于采集层和处理层之间的高性能无锁/异步通道传递。 |
| **元数据存储** | `rusqlite` (SQLite) / `clickhouse-rs` | 原型阶段用 SQLite 快速落地；生产环境建议切换为 ClickHouse 处理海量数据。 |
| **AI / LLM 接入** | `reqwest` (HTTP) / `ollama-rs` | 调用云端大模型 API，或通过 `ollama-rs` 调用本地部署的开源模型（保证隐私）。 |
| **CLI / TUI** | `clap` (参数解析) / `ratatui` (终端 UI) | `clap` 处理命令行参数；`ratatui` 构建类似 `htop` 的实时终端仪表盘。 |
| **序列化** | `serde` / `serde_json` / `bincode` | 高效的数据序列化与反序列化。 |

## 5. 非功能需求 (Non-Functional Requirements)

1.  **性能指标**：
    *   在普通千兆网卡下，抓包丢包率 < 0.1%。
    *   协议解析与 DPI 分类延迟 < 1ms/包。
2.  **内存安全**：
    *   充分利用 Rust 的所有权机制。
    *   **关键点**：在调用 nDPI (C 库) 时，必须严格封装 `unsafe` 代码块，确保内存分配和释放的绝对安全，防止内存泄漏或 UAF（释放后使用）。
3.  **可扩展性**：
    *   协议解析器需设计为 Trait 接口，方便未来添加新的自定义协议解析。
4.  **隐私与合规**：
    *   在将数据发送给外部 LLM API 前，必须经过正则脱敏（如抹除 MAC 地址、内网 IP、敏感 Payload）。建议默认使用本地 LLM（如 Llama-3/Qwen）。

## 6. 原型开发路线图 (MVP 阶段划分)

为了避免项目烂尾，建议采用敏捷迭代，分为四个阶段：

### Phase 1: 基础骨架 (The "tcpdump" phase) - *预计 2 周*
*   **目标**：能跑起来，能抓包，能看基础信息。
*   **任务**：
    *   使用 `clap` 搭建 CLI 框架。
    *   使用 `pcap` 和 `etherparse` 实现基础抓包和五元组解析。
    *   实现终端实时打印抓包信息（类似 tcpdump 输出）。

### Phase 2: 深度解析与分类 (The "DPI" phase) - *预计 3 周*
*   **目标**：知道这到底是什么流量。
*   **任务**：
    *   引入 `nDPI` FFI 绑定，实现应用层协议识别。
    *   实现 TCP 流重组。
    *   将解析后的元数据异步写入 SQLite 数据库。

### Phase 3: AI 接入与交互 (The "AI" phase) - *预计 3 周*
*   **目标**：让工具“听懂”人话。
*   **任务**：
    *   接入 LLM API（或本地 Ollama）。
    *   实现 Text-to-BPF/SQL 功能（核心 Prompt 工程）。
    *   实现基于元数据的流量总结功能。
    *   使用 `ratatui` 开发一个基础的 TUI 面板，展示实时流量统计和 AI 对话窗口。

### Phase 4: 优化与高级特性 (The "Polish" phase) - *预计 2 周*
*   **目标**：提升性能，增加高级 AI 功能。
*   **任务**：
    *   性能调优（减少内存分配，优化 zero-copy）。
    *   实现 TLS JA3 指纹提取与 AI 恶意软件匹配。
    *   实现原始 Pcap 文件的切片存储与索引回溯。

## 7. 风险与应对策略

| 风险点 | 影响 | 应对策略 |
| :--- | :--- | :--- |
| **nDPI FFI 内存泄漏/崩溃** | 导致整个 Rust 程序 Panic 或内存暴涨。 | 严格限制 `unsafe` 代码范围；编写大量的 FFI 边界测试；使用 `valgrind` 或 `miri` 进行内存检查。 |
| **LLM 响应延迟导致 UI 卡顿** | 用户体验极差。 | 采用异步架构，AI 请求放入后台 Task；TUI 界面采用非阻塞设计，AI 结果以流式（Streaming）或弹窗形式异步展示。 |
| **海量数据导致 SQLite 写入瓶颈** | 丢包或数据丢失。 | 引入批量写入（Batch Insert）机制；使用内存 WAL 模式；若数据量极大，及时切换至 ClickHouse。 |
| **AI 幻觉导致误报** | 误导用户。 | 明确 AI 的定位是“辅助分析”而非“自动拦截”；所有 AI 给出的结论必须附带原始数据证据（如“根据以下 3 个包的 Payload 判断...”）。 |

---

## 8. 下一步行动建议

1.  **环境搭建**：安装 Rust 工具链，配置 `libpcap-dev` (Linux) 或 `WinPcap/Npcap` (Windows) 开发环境。
2.  **PoC 验证**：不要一开始就写大框架。先写一个 100 行的 Rust 脚本，用 `pcap` 抓包并用 `etherparse` 打印出 IP 和端口，验证底层链路。
3.  **nDPI 绑定调研**：评估是直接使用现有的 Rust nDPI 绑定（如果有且维护良好），还是自己用 `bindgen` 生成。

这份 PRD 为你提供了一个清晰、可落地的 Rust 开发蓝图。Rust 的强类型和内存安全机制会在后期处理复杂网络协议和 FFI 调用时为你省去大量 Debug 的时间。祝开发顺利！
